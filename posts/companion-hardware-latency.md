做一个会聊天的毛绒玩具,接一个大模型 API 只是开始。真正的工程问题是:**用户对「活物」的延迟容忍度只有约 800ms**,而你要在这个预算内完成听、想、说三件事。

## 延迟预算的拆解

```text
唤醒检测(端侧)      ~0ms    WakeNet 常驻,不计入对话延迟
语音识别(流式)      150ms   首字延迟,边说边转
LLM 首 Token         300ms   模型路由:简单意图走小模型
TTS 首音频帧         200ms   流式合成,首句优先
网络与缓冲           150ms   国内链路,留余量
────────────────────────────
合计                 ~800ms
```

每一项都有端云分配的选择题:

- **唤醒**必须在端侧——这是隐私合规的底线,也是功耗的现实(云端常听不可行)
- **ASR** 流式上云,端侧只做 VAD 切分
- **LLM** 做意图路由:问候、撒娇类高频意图走轻量模型,复杂对话才升级到大模型
- **TTS** 流式返回,首句到达即播放,后续句子边合成边缓冲

## 「灵魂感」来自填充延迟,而不是消灭延迟

800ms 在物理上很难再压缩,但体验上可以**用动作填充**:收到唤醒后立即让玩具做出「歪头/耳朵动」的反应,用户感知到的响应是 0ms,真正的语音回复在动作期间到达。

> 人类对话中的停顿本来就有 200–600ms。问题不是延迟本身,而是延迟期间的「死寂」。

## 成本侧的硬约束

每次对话的云端成本(ASR + LLM + TTS)决定了商业模式。粗算:如果日均对话 40 轮,单轮云成本必须压到 1 分钱以下,订阅制才能跑通。这反过来决定了模型路由策略——**路由不是性能优化,是商业模式的一部分。**
